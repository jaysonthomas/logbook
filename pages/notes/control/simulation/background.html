Background

Overview

Simulation is all about advancing a system's state forward in time. A discrete-time simulation helps simulate continuous dynamics (represented by ODEs which are always continuous), digitally on a PC.

Euler integration is based on approximating the next discrete state by linearising the continuous dynamics using Taylor series expansion.

Caution

Sanity check behaviours (energy, momentum behaviour etc) where the outcome is already known. Never trust a simulation at face value.

Motivation

  • It is only possible to solve $\dot{x}=f(x)$ for $x(t)$, for linear systems and very simple non-linear ones.
  • Digital devices can't represent continuous states $x(t)$. Discrete states $x_k$ are required, instead.
  • Discrete time models can capture some things that continuous ODEs cannot, like contact-impacts that are non-differentiable.

    • When a brick is dropped, its velocity goes from some non-zero value to 0 instantaneously, the moment it hits the ground. The acceleration goes to infinity. Therefore, it is not possible to represent such discontinuous jumps with an ODE. In discrete time, it would be represented as a simple instantaneous jump.

System dynamics

To implement a function that advances a system's state forward in time, its dynamics (the impact of forces on an object's motion) needs to be understood/modelled.

A system's dynamics can be abstracted as a plant that will output the next state $x$, given the torque applied $\tau$ and the current state. This abstraction permits a system viewpoint.

System output

The output of a system doesn't have to be its full state $x$ - it can be a subset of the state obtained via some sensor attached to the system! Where there is full access to the state for feedback, i.e. full-state feedback, the system becomes more deterministic and the future value of the system can be predicted given the current time, state, and input to the system.
