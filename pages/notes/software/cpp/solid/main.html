SOLID design principles

Overview

Single Responsibility

A class should have only one reason to change.

However, classes shouldn't be too specific leading to 100s of ridiculously interconnected classes, that could easily be one. SRP is about limiting the impact of change - group together things that change for the same reasons and separate those that change for different reasons. There should be a good balance.

One indication that SRP is not followed is when a constructor has more than 5-6 parameters.

Open/Closed

The principle dictates the software modules are open for extension but closed for modification. This allows new functionality to be added without changing existing source code.

Polymorphism is the best way to achieve this - with the use of abstract interfaces, we can extend a class and specialise its behaviour without making changes to the parent. This principle allows for reusable and maintainable software.

Liskov substitution

Subtypes must be substitutable for their base types without altering the correctness of the program. This states that a parent class (say when passed as an argument to a function) should be substitutable by any of its derived classes. This principle ensures that inheritance is used correctly.

Interface Segregation

Clients should not be forced to depend on interfaces that they do not use. This principle advocates for the creation of small and atomic interfaces. Abstractions/interfaces should be designed in such a way that the clients that are using the exposed methods do not have to get the whole pie. Clients should not have to carry the burden of implementing methods that they don't actually need.

Dependency of inversion

  • High-level modules should not depend on low-level modules. Both should depend on abstractions.
  • Abstractions should not depend on details. Details should depend on abstractions.
High-level modules orchestrate low-level modules in an application. Whilst low-level modules are more specific to components that are then used by the high-level modules.
